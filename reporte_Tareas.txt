2. SPRINT RETROSPECTIVE (RETROSPECTIVA)
--------------------------------------------------------------------

¿Qué hicimos bien? (Madurez del equipo):
1. Alto enfoque en los módulos core: Se consolidó de forma excelente el CRUD y ciclo de vida de Clientes, Vehículos y Diagnósticos, sentando bases sólidas para el sistema.
2. Gran capacidad de estimación inicial: Se abarcó un volumen ambicioso de requerimientos con un desglose claro de pesos en tallas (XS a L) alineados con la escala Fibonacci.
3. Desacoplamiento efectivo: El avance paralelo de las funcionalidades demuestra buena arquitectura inicial.

¿Qué podemos mejorar? / Problemas detectados:
1. Gestión de Rutas y Navegación: El principal cuello de botella ocurrió en la definición y comunicación de las rutas del sistema (API/Frontend Router), bloqueando la entrega final de 6 historias críticas vinculadas a las órdenes de trabajo.
2. Criterios de Aceptación y Pruebas Tempranas: Los problemas de rutas debieron detectarse antes en el ciclo mediante pruebas de integración tempranas en lugar de acumularse al cierre del sprint.

Acciones de Mejora para el Sprint 2:
1. Sesión de Refactorización de Rutas: Dedicar las primeras horas del Sprint 2 a resolver de manera definitiva la arquitectura de enrutamiento para desbloquear las HU-20, HU-21, HU-23, HU-24, HU-25 y HU-26.
2. Implementar Mocking o Contratos de API Claros: Definir los endpoints antes de codificar la interfaz para asegurar que el enrutamiento no sea un stopper.

Principios SOLID:
S-Cada clase tiene una sola responsabilidad.
Usuario.js solo maneja usuarios, Cliente.js solo maneja clientes.

O- Abierto/Cerrado
Class abiertas para extensión, cerradas para modificación.
Los modelos heredan de pool y pueden extenderse sin modificar la clase base.

L-Sustitución de Liskov:
Las subclases pueden sustituir a sus clases base.
Los controladores usan req y res de Express de manera consistente.

I - Segregación de Interfaces:
Interfaces específicas en lugar de una general.
verificarToken y verificarRol son middlewares separados.

D - Inversión de Dependencias:
Depender de abstracciones, no de concretas.
Los modelos dependen de pool (abstracción de base de datos).

Patrones Creacionales

Patron Singleton
Propósito: Asegurar que solo exista una instancia de una clase.
javascript
// src/config/db.js - Única instancia del pool
const pool = new Pool({ ... });
module.exports = pool; // ← Solo una instancia en toda la app


Patrones Estructurales

Patrón DTO (Data Transfer Object):
Propósito: se usa para transportar datos entre diferentes capas de una aplicación, sin exponer la lógica interna o la estructura completa de los objetos.
El frontend solo recibe lo que necesita, sin datos sensibles.

// controllers/authController.js
res.json({
  success: true,
  token,
  usuario: {
    id: usuario.id,
    nombre_completo: usuario.nombre_completo,
    nombre_usuario: usuario.nombre_usuario,
    correo: usuario.correo,
    rol: usuario.rol
  }
});
Patrones de Comportamiento
Middleware (Variante de Chain of Responsibility)
Propósito: Procesar peticiones antes de que lleguen al controlador.
Separa la autenticación de la lógica de negocio.
javascript
// middlewares/auth.js
const verificarToken = (req, res, next) => {
  // Procesa el token
  if (!token) return res.status(401).json(...);
  // Si pasa, llama a next() para continuar
  next();
};
